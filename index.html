Step 1: Basic idea

Imagine a smart school office:

  • Router Agent is the receptionist. It decides which expert should help.
  • GTM Agent answers product and pricing questions using your docs and web context.
  • Outreach Agent writes marketing content, finds leads, and can send emails if user intent is explicit.

The system passes around one shared state object called AgentState with fields like question, context, answer, and trace steps.

Step 2: Live graph of your actual pipeline

normal edge dashed = conditional route

Child version: the receptionist picks a hallway (GTM or Outreach). Each hallway has checkpoints before final answer. Some checkpoints can end early.

GTM lane (product + pricing)

  • gtm_retrieve: gather context from Qdrant docs + web.
  • pricing_gate: asks "is this pricing?"
  • If yes, collect_email enforces email gate.
  • gtm_generate writes final grounded answer.

Outreach lane (content + sending)

  • outreach_research: uses Apollo for leads, else KB + web.
  • outreach_generate: drafts post/email from context.
  • send_gate: send now vs review-only decision.
  • outreach_send: extracts emails and sends via SendGrid.

Step 4: Tools the agents can use

  • search_knowledge_base(query)
  • web_search(query)
  • apollo_search(job_titles,...)
  • send_email(to, subject, html)

Step 5: State fields

  • question: user input
  • agent_type: gtm or outreach
  • context: tool outputs
  • answer: final response
  • steps: trace list (appended across nodes)

Step 6: Evals overview

  • 17 prompts across GTM, outreach, send, leads, adversarial, mixed.
  • Tracks expected tool coverage in sessions mode.
  • Can run as Galileo experiment with env switch.

Step 7: End-to-end in one sentence

User asks question -> Router chooses expert -> expert collects context with tools -> gates handle safety/business rules (pricing email or send intent) -> expert generates answer -> optional send executes -> every step is logged for traces/evals.
